What it does
A user points IssueForge at any GitHub repo and a specific issue. The system autonomously reads the codebase, understands the required change, writes a patch, runs the test suite, critiques its own output, and opens a pull request — with no human intervention between issue and PR.

Each specialist agent is purpose-built: Explorer maps the relevant files using AST analysis and semantic search, Planner produces a dependency-ordered task graph, Coder iterates on patches with a sandbox code executor, and Reviewer runs tests and applies a constitutional self-critique before the PR is submitted.
What makes it a showcase piece
It's benchmarkable — you run it against SWE-bench Lite (300 real GitHub issues from popular repos) and publish your resolve rate. Even a 10–15% resolve rate puts you in credible territory; top open-source systems are at 40–50%. Your README shows the number, which is an immediately legible signal of capability.

It's built from scratch — no LangChain, no CrewAI. Recruiters and collaborators can read your agent loop, tool registry, and memory store directly. The absence of framework magic is the point.

It's practically useful — run it as a GitHub Action on your own repos. Every time you merge a bug fix it produced, that's a real testimonial.
Tool surface each agent needs
Explorer: read_file, list_dir, grep, ast_parse, vector_search
Planner: read_task_context, write_task_graph
Coder: write_file, apply_patch, run_command (sandboxed)
Reviewer: run_tests, read_diff, approve_or_reject
Orchestrator: all of the above, plus open_pr (GitHub API)
Each capstone week maps to Phase 4 of the curriculum, but calls back skills built in Phases 1–3. Build in order — each week's deliverable is the input to the next.
Week 10
Core abstractions + Explorer + Planner
Finalize your Agent base class, Tool registry, and Memory interface from Phase 1–2 work · Build Explorer: AST traversal (tree-sitter), ChromaDB semantic index of repo, grep + file read tools · Build Planner: takes Explorer's context map + issue text, produces a JSON task DAG · Wire Orchestrator shell with single-turn delegation
Deliverable: given an issue, system can identify the right files and produce a plausible task list
Week 11
Coder + Reviewer + Memory integration
Build Coder: patch generation loop with sandboxed run_command (Docker subprocess), test-driven iteration (run → observe failure → revise), max 5 retries with backoff · Build Reviewer: constitutional self-critique prompt, must pass linting + original test suite · Connect Memory store: ChromaDB persists codebase embeddings + past issue resolutions · Add human-in-the-loop checkpoint before PR open · Wire full Orchestrator: issue → explore → plan → code → review → PR
Deliverable: end-to-end run on a simple bug issue in a test repo
Week 12
Eval, observability, polish, postmortem
Run against SWE-bench Lite subset (start with 30 issues): record resolve rate, token cost per task, latency · Add OTEL tracing: span per agent per tool call, cost dashboard · Optimize: route simple issues to a smaller model (claude-haiku), cache codebase embeddings · Write the README: architecture diagram, benchmark results, quick-start, contributing guide · Write the postmortem: what failed, what surprised you, what you'd build next
Deliverable: public GitHub repo with benchmark score, full tracing, documented postmortem

GitHub Action packaging
Wrap IssueForge as a GitHub Action: developers add it to their repo workflow, label any issue with issueforge, and a PR appears within minutes. Zero-friction adoption — the agent runs in their CI environment using their own API key.
SWE-bench as credibility
The resolve rate is a legible, comparable number. A README badge showing "resolves X% of SWE-bench Lite" puts IssueForge next to commercial systems (Devin, SWE-agent, Agentless). Even 12–15% makes it a credible research artifact worth starring.
Path to SaaS
Add a thin FastAPI layer, GitHub OAuth, and a per-repo subscription. The core value prop is clear: developers pay per resolved issue. Pricing is easy to justify — if it fixes a 2-hour bug in 5 minutes, the economics are obvious.
Open source traction
Publish with a detailed architecture writeup, benchmark results, and the postmortem. The ML/agent community rewards "built-from-scratch + benchmarked" projects. Submit to arXiv as a technical report — the bar for 2–3 page system papers is low and the visibility is high.
Differentiation from existing tools
SWE-agent and Agentless use single-model approaches. IssueForge's value is the multi-agent specialization — Explorer's semantic code map makes Coder's patches more precise than vanilla file-read-and-edit. Your postmortem can directly compare architectures, which is the kind of analysis that gets cited.
